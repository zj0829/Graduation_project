# 4.3 工具执行层实现

工具执行层是本系统的底层基础支撑模块，负责具体安全测试操作的落地执行。该层采用Docker容器化技术将各类异构安全测试工具封装为标准化执行单元，通过自研的MCP（Mission Control Protocol）智能调度网关实现工具的统一注册、参数化调用与结果格式化输出。工具执行层的设计核心在于"适配器模式"的深度应用——通过统一的接口抽象屏蔽不同工具在命令语法、输入输出格式、运行环境等方面的差异，使上层调度控制层无需关心底层工具的具体实现细节。

## 4.3.1 MCP服务架构与集成

### （1）MCP协议定制与实现

本系统并未采用第三方MCP客户端（如CherryStudio），而是基于JSON-RPC 2.0协议自主设计并实现了专用的MCP智能调度网关（MCP Gateway）。该网关采用Node.js Express框架开发，监听于9876端口，对外暴露标准的JSON-RPC接口，支持三种核心方法：

- **initialize**：建立客户端会话，返回服务端能力声明（server capabilities）和会话标识符（session_id），用于关联后续所有请求
- **tools/list**：返回当前已注册的全部可用工具清单，包含工具名称、功能描述、所属分类、参数定义等元信息
- **tools/call**：根据指定的工具名称和参数字典，路由至对应的工具执行函数，返回结构化的执行结果

MCP网关的核心数据结构为TOOLS_CONFIG配置对象，以工具名称为键，每个工具条目包含name（显示名）、description（功能描述）、category（分类标签）、container（容器标识）、command/ apiUrl/rpcUrl等连接信息，以及parameters参数数组（定义每个参数的名称、类型、是否必填、说明文字）。这种配置驱动的架构使得新增工具仅需在配置对象中添加条目并实现对应执行函数即可完成集成，无需修改协议处理逻辑。

### （2）通信机制

后端FastAPI服务通过httpx异步HTTP客户端与MCP网关进行通信。考虑到本地部署环境下IPv6地址解析可能导致连接异常的问题，通信地址统一使用127.0.0.1:9876而非localhost。每次测试任务发起时，后端按以下固定流程与MCP网关交互：

首先发送initialize请求建立会话上下文；随后通过tools/list获取可用工具列表，供LLM决策参考；最后根据LLM生成的测试计划或预设的工具策略，依次发送tools/call请求触发各工具的执行。整个交互过程采用异步非阻塞方式，单个工具的超时时间根据工具特性差异化设置（Nmap设为5分钟，Amass设为10分钟，SQLmap设为10分钟），确保长时间运行的扫描任务不会被过早中断。

### （3）日志与监控

MCP网关内置了双通道日志机制：一方面通过console.log向终端输出实时运行状态（用于调试）；另一方面通过fs.appendFileSync将所有操作记录以UTF-8编码写入data/mcp.log文件（用于事后审计）。每条日志包含北京时间戳（Asia/Shanghai时区）、来源模块标记和详细内容。此外，网关还提供了/health健康检查端点和/sse服务器推送事件端点作为备用监控手段。

## 4.3.2 工具调用流程

一次完整的工具调用流程包含以下四个阶段：

**阶段一：会话初始化**。后端向MCP网关发送initialize请求，携带客户端标识（client字段设为"ai-chat"或"LLM-Penetration-Testing"）和版本号。MCP网关响应中返回sessionId、server信息（含服务名、版本号、可用工具数量）和capabilities声明。该阶段建立了本次测试任务的逻辑会话边界。

**阶段二：工具发现**。后端调用tools/list方法，MCP网关遍历TOOLS_CONFIG配置对象，返回全部7个已注册工具的元信息数组。每个工具对象包含id、description、category和parameters等字段。这些信息随后被注入到LLM的提示词中，作为AI决策的参考依据。

**阶段三：工具执行**。后端根据测试策略确定待调用的工具列表。系统支持三种策略模式：（1）auto模式——自动选择Nmap+SQLmap作为默认组合；（2）全工具模式——依次调用所有已注册工具；（3）自定义模式——由用户或LLM指定工具子集。对于每个工具，后端构造tools/call请求，MCP网关根据tool字段路由至对应的执行函数。

**阶段四：结果处理**。工具执行函数返回的结果经过标准化处理后回传给后端。标准结果格式包含status（成功/失败）、timestamp（北京时间戳）、result对象（内含raw_output原始输出、parsed_data解析后的结构化数据、execution_time执行时间等字段）。若工具执行异常，则返回error对象包含错误码和错误描述。

## 4.3.3 安全测试工具集成

本系统通过Docker容器化技术集成了6类共7个专业安全测试工具，所有容器均接入名为pentest-network的自定义bridge网络，实现容器间的互通互联。各工具的具体集成方案如下：

### （1）Nmap——网络扫描与资产发现

- **镜像**：instrumentisto/nmap:latest
- **容器名**：pentest-nmap
- **启动配置**：覆盖entrypoint为空，主进程执行sleep infinity保持容器常驻运行
- **调用方式**：docker exec pentest-nmap nmap [options] [target]
- **默认参数**：-sV -sC（版本探测+脚本扫描）
- **输入预处理**：自动剥离目标URL中的http(s)://前缀和路径部分，仅保留主机名或IP
- **输出解析**：正则匹配提取开放端口号、协议类型、服务名称和版本信息，组装为结构化端口列表
- **超时设置**：300秒（5分钟）
- **容错机制**：当Docker不可用时降级为本地nmap命令执行；当工具完全不可用时返回模拟数据（含4个常见端口的模拟扫描结果）

### （2）Amass——子域名枚举与资产测绘

- **镜像**：caffix/amass:latest
- **容器名**：pentest-amass
- **启动配置**：同Nmap，sleep infinity常驻运行
- **调用方式**：docker exec pentest-amass amass enum [options] -d [domain]
- **默认参数**：-passive（被动枚举模式，减少对目标的影响）
- **输出解析**：从原始输出中提取发现的子域名条目，去重后返回子域名列表
- **超时设置**：600秒（10分钟）
- **应用场景**：目标为域名时的资产面扩展，发现隐藏的测试环境和未公开的服务实例

### （3）OWASP ZAP——Web应用漏洞自动化扫描

- **镜像**：zaproxy/zap-stable:latest
- **容器名**：pentest-owasp-zap
- **启动配置**：以守护进程模式（daemon）启动，绑定8090端口到宿主机，开放API访问权限（api.addrs.addr.regex=true），禁用API密钥验证以便程序化调用；JVM内存限制为512MB
- **调用方式**：通过ZAP REST API（http://pentest-owasp-zap:8090）进行交互
- **支持的扫描类型**：
  - spider（爬虫）：自动发现目标站点的URL链接结构
  - active_scan（主动扫描）：对发现的URL执行各类攻击payload检测
  - passive_scan（被动扫描）：分析代理流量中的潜在安全问题
- **输出解析**：从ZAP API获取告警列表（alerts），提取告警ID、风险等级（High/Medium/Low/Info）、漏洞名称、URL位置、参数信息和修复建议，限制返回前20条最高风险告警
- **特殊处理**：ZAP是唯一不以exec方式调用的工具，而是通过RESTful API进行远程控制，体现了适配器模式的灵活性

### （4）SQLmap——SQL注入自动化检测与利用

- **镜像**：pentest-sqlmap（基于sqlmap官方镜像自建，Dockerfile位于tools/sqlmap/目录）
- **容器名**：pentest-sqlmap
- **启动配置**：sleep infinity常驻运行
- **调用方式**：docker exec pentest-sqlmap python3 sqlmap.py -u [url] [options]
- **默认参数**：--random-agent（随机User-Agent规避WAF检测）
- **输出解析**：从原始输出中识别注入点参数、数据库类型、可用的注入技术和提取的数据库名/表名等信息
- **超时设置**：600秒（10分钟）
- **安全特性**：使用--level和--risk的默认保守值，避免对目标造成破坏性影响

### （5）Metasploit Framework——综合漏洞利用与后渗透

- **镜像**：metasploitframework/metasploit-framework:latest
- **容器名**：pentest-metasploit
- **依赖组件**：pentest-metasploit-db（PostgreSQL 15-alpine，存储Metasploit的会话数据和漏洞库信息）
- **端口映射**：55553（MSF-RPC）、4444（MSF-Web）
- **启动配置**：覆盖entrypoint为空，常驻运行；通过环境变量MSF_DATABASE_CONFIG配置PostgreSQL连接串
- **调用方式**：docker exec pentest-metasploit /usr/src/metasploit-framework/msfconsole -q -x "use [module]; setg [options]; run; exit"
- **参数传递**：模块路径（module）和选项键值对（options对象）通过命令行拼接传入msfconsole的非交互式批处理模式
- **输出处理**：捕获msfconsole的标准输出和错误流；当命令执行抛出异常时，优先使用stdout中的有用输出作为结果
- **超时设置**：300秒（5分钟）

### （6）Burp Suite——Web安全辅助测试平台

- **镜像**：portswigger/burp（Community Edition）
- **容器名**：pentest-burpsuite
- **启动配置**：Headless模式下运行，提供Web界面访问入口
- **调用方式**：Burp Suite主要通过其Web界面进行人工操作，系统提供容器状态检查和URL入口返回
- **定位**：作为半自动化辅助工具存在，适用于需要人工判断的高级安全测试场景
- **特殊说明**：与其他全自动化工具有所区别，Burp Suite在本系统中扮演"人机协作节点"的角色

## 4.3.4 工具适配器设计

工具适配器是工具执行层的核心抽象组件，负责桥接上层标准化指令与底层异构工具之间的语义鸿沟。本系统的适配器设计遵循以下原则：

### （1）统一接口契约

所有工具执行函数共享相同的函数签名模式：async function executeXxx(params) → Promise<Object>。输入为参数字典（key-value形式），输出为包含raw_output（原始文本）、parsed_data（结构化数据）、execution_time（执行时间戳）等字段的标准化结果对象。无论底层工具是命令行工具（Nmap、Amass、SQLmap、Metasploit）还是REST API服务（OWASP ZAP），对外呈现一致的接口形态。

### （2）参数转换层

适配器负责将MCP协议中的通用参数映射为各工具特有的命令行格式或API请求格式。例如：
- Nmap的target参数经URL预处理（去除协议头和路径）后直接追加至命令末尾
- SQLmap的url参数保持原样传给-u标志
- Metasploit的options对象被展平为setg key=value格式的命令序列
- OWASP ZAP的scan_type参数决定调用spider还是ascan API端点

这种参数转换逻辑封装在各自的执行函数内部，对调用方透明。

### （3）执行环境管理

适配器统一管理工具的执行环境。对于Docker容器化工具，适配器首先探测Docker守护进程的可用性（通过docker ps命令），若Docker可用则使用docker exec在指定容器内执行命令；若Docker不可用则降级为本地原生命令执行。这种双轨制执行策略保证了系统在容器环境不完整时仍能提供基本功能（模拟数据模式）。

执行环境的资源约束通过三个维度控制：
- **timeout**：单次执行的最大允许时长（5~10分钟不等）
- **maxBuffer**：标准输出的最大缓冲区容量（50~100 MB）
- **encoding**：统一使用UTF-8编码处理工具输出，确保中文和多语言内容的正确解析

### （4）输出解析与标准化

各工具的原始输出格式差异巨大：Nmap输出类表格文本、Amass输出纯域名列表、ZAP输出嵌套JSON、SQLmap输出混合状态栏和结果表、Metasploit输出交互式终端文本。适配器为每种工具实现了专门的输出解析器（parseXxxOutput系列函数），将异构的原始输出转换为统一的内部数据格式。解析后的数据包含：
- 结构化字段（如端口列表、子域名列表、漏洞列表）
- 原始文本保留（raw_output）用于报告生成时的原文引用
- 元数据（执行时间、目标信息、使用的参数）用于审计追溯

### （5）错误处理与降级策略

适配器实现了多层次的容错机制：

**第一层**：try-catch包裹整个执行过程，捕获同步异常（如命令不存在、参数非法）和异步超时异常，将错误信息包装为标准错误响应返回。

**第二层**：针对特定工具的已知问题提供友好提示。如Amass执行失败时建议检查域名格式和容器状态；Nmap在Docker不可用时提供模拟扫描数据。

**第三层**：全局JSON-RPC错误码规范。当工具名称不在TOOLS_CONFIG中时返回method not found错误（隐含于switch-default分支）；当工具执行过程中抛出未预期异常时，返回internal error（code: -32603），携带具体的错误消息。

通过以上五层设计，工具适配器实现了对7种异构安全测试工具的标准化管理，使得上层调度控制层只需关注"调用什么工具"和"传入什么参数"，而无需关心"如何调用"和"如何解析结果"的实现细节。这种设计显著降低了新工具集成的技术门槛——添加一个新工具通常只需要：(1) 在TOOLS_CONFIG中添加配置条目；(2) 实现一个executeXxx函数；(3) 实现对应的parseXxxOutput解析函数；(4) 在handleToolsCall的switch-case中添加路由分支。
